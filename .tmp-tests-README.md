# TFP Test System Guide

## 1. Overview

This test system is organized by **test type first** and **business domain second**.

- `apps/api/tests` is the canonical location for API unit/integration tests.
- `apps/web/tests` is the canonical location for web unit tests.
- `tests/e2e` covers real browser behavior and SSR flows with Playwright.
- `tests/contract` covers deterministic non-browser validation (currently i18n contracts).
- `tests/create-data` is the canonical browser-seed SSOT (assets, runner configs, contracts, manifests).
- `tests/seed` remains as a compatibility source during migration.
- `tests/manual` holds human-run QA playbooks.
- `tests/fixtures` stores reusable static assets used by tests.

Why this structure:

- keeps real user journeys close to their domain,
- avoids phase-based sprawl,
- preserves fast deterministic checks where browser coverage is unnecessary,
- makes ownership and extension paths obvious for future contributors.

## 3. Test-Type Rules

- Use `tests/e2e/domains` for feature/domain-specific browser behavior.
- Use `tests/e2e/journeys` for multi-domain user journeys.
  - `tests/e2e/journeys/tests/scenario-s-shared-detail-tabs-and-submissions.spec.ts` validates shared detail tabs + submissions behavior across contests, events, and opportunities.
- Use `tests/e2e/a11y` for accessibility-focused browser checks.
- Use `tests/e2e/smoke` for fast launch-gate health checks.
- Use `tests/e2e/visual` for screenshot/visual guardrails.
- Use `tests/contract` for deterministic structural rules and schema/content contracts.
- Use `tests/create-data` as the canonical UI-seed config/assets source used by Playwright browser creation flows.
- Use `tests/manual` only for flows that are not yet practical to automate.

## 4. Domain Rules

Domain placement for E2E specs:

- Auth: `tests/e2e/domains/auth/tests`
- Contests: `tests/e2e/domains/contests/tests`
- Opportunities: `tests/e2e/domains/opportunities/tests`
- Events: `tests/e2e/domains/events/tests`
- Profile: `tests/e2e/domains/profile/tests`
- Moderation: `tests/e2e/domains/moderation/tests`
- Admin: `tests/e2e/domains/admin/tests`
- Localization: `tests/e2e/domains/localization/tests`
- Uploads/media: `tests/e2e/domains/uploads/tests`
- Notifications: `tests/e2e/domains/notifications/tests`
- SEO/public metadata/system constraints: `tests/e2e/domains/seo/tests`, `tests/e2e/domains/system/tests`

If a flow spans multiple domains, put it in `tests/e2e/journeys/tests`.

## 5. Helper Rules

- Put cross-domain helpers in `tests/e2e/shared/helpers`.
- Default E2E spec imports should come from `tests/e2e/shared/fixtures/e2e.fixture.ts` so console/network failure capture is installed consistently.
- Keep domain-specific flows/assertions local to the domain test file unless reused by 2+ specs.
- Keep fixtures in `tests/e2e/shared/fixtures` only when shared broadly.
- Keep page objects in `tests/e2e/shared/pom` when they model stable reusable UI shells.
- Keep setup and seeded account constants in `tests/e2e/shared/setup` as single sources of truth.

## 6. Fixtures and Assets

- Use `tests/fixtures/media` for reusable upload/image fixtures.
- Use `tests/fixtures/uploads` for upload payload variants.
- Use `tests/create-data/assets/**` when the fixture is intentionally tied to browser seed profiles.
- Add domain-local assets only when they are truly domain-specific and unlikely to be reused.
- Naming:
  - prefer descriptive names (`valid-avatar.jpg`, `oversized-banner.jpg`),
  - avoid generic `test1.jpg` / `sample.png` names,
  - keep lowercase kebab-case.
- Avoid duplicate copies of the same fixture across domains.

## 7. Selector Strategy

- Prefer semantic locators: `getByRole`, `getByLabel`, `getByText` with stable wording.
- Prefer app-level stable attributes where available (for example data attributes used intentionally for testing).
- Avoid brittle deep CSS selectors tied to style classes.
- Use deterministic waits (`waitForURL`, explicit target states) instead of arbitrary sleep timers.

## 8. Data Strategy

- Deterministic E2E baseline is controlled by:
  - `tests/e2e/shared/setup/global-setup.mjs` (reset and seed launcher)
  - `tests/e2e/shared/setup/seeded-accounts.ts` (seeded account SSOT)
  - `playwright.config.ts` (suite-level web server + setup wiring)
- Prefer browser-first assertions in `tests/e2e/**`; use direct DB writes only for narrow seed/setup gaps that cannot yet be expressed safely through public or test-only APIs.
- Seeded usernames are part of the SSOT (`admin-user`, `test-photographer`, `test-model`); use username profile routes (for example `/profile/test-photographer`) instead of email-encoded profile paths.
- Use seeded test identities for auth/role coverage; do not create random account scaffolding in core regression suites unless the test explicitly requires it.
- Keep i18n contract checks deterministic and locale-scoped; prefer env-driven locale subsets for faster local runs.
- Preserve proper nouns (for example `TFP`, `TFP Photographers`) and placeholder parity in locale catalogs.

Process lifecycle SSOT:

- `scripts/stop-app.sh` is the canonical stop primitive for local FE/BE cleanup.
- `scripts/restart-app.sh` and QA session scripts (`scripts/qa/prepare-session.sh`, `scripts/qa/restart-stack.sh`) must delegate to `scripts/stop-app.sh` instead of duplicating process-kill logic.
- Canonical runtime environments are now `local`, `test`, `development`, and `production`.
- `local` uses the local filesystem storage adapter and the `tfp_photographers_local` database.
- `test` reuses the local filesystem adapter but isolates destructive automated test runs in `tfp_photographers_test` so local manual QA data survives `pnpm test*` and coverage flows.
- `development` uses the Backblaze/B2 adapter and the `tfp_photographers_dev` database.
- `production` uses production config defaults and the `tfp_photographers_prod` database when pointed at local Postgres URLs.
- For moderation/admin manual QA, prefer `pnpm qa:seed:all:quick`. Use `pnpm qa:db:hard-reset:admin` first if you want a clean DB baseline. The flow seeds approved, flagged, rejected, and pending-AI fixtures, including queued moderation jobs for the `/admin?tab=pending` tab.
- Browser seed stack restarts apply pending Prisma migrations before the FE/BE processes start, so newly added seed fields are available before Playwright creates content.
- The quick moderation profile now also creates deterministic event text-moderation fixtures titled `Text moderation 1`, `Text moderation 2`, and `Text moderation 3` so local admin review includes spam/contact and explicit-text cases with clean cover images.
- `Text moderation 1` is intentionally an off-platform contact-spam fixture (`WhatsApp + phone number`) and should now resolve to an AI text rejection instead of a review-only flag.
- The browser seed login verifier now retries protected-page session checks after sign-in, which makes admin handoff between the seeded member accounts and `admin@tfp.local` more reliable during `qa:seed:all:quick`.
- The browser seed login flow now retries one invalid-credential pass before failing, which stabilizes owner re-login during the moderation quick profile when freshly created users are revisited later in the same run.
- Contest submission interaction waits for the moderation-approved entry to appear on the submissions tab before triggering the seeded reaction flow.
- Local browser seed runs now take an exclusive lock in `tmp/qa/seed-browser-flow.lock.json`, so overlapping `qa:seed:*` runs fail fast instead of resetting the same local DB and stack underneath each other.
- When you run a destructive DB reset while the local FE/BE stack is already running, restart the stack before browser moderation validation. This avoids stale worker/query state during admin moderation review passes.
- Event and contest detail regressions should include manual viewport checks on mobile, tablet, and desktop, plus event photo-upload validation:
  - event owner and attendee limits come from `MEDIA_SUBMISSION_LIMITS.event` in `packages/shared/src/submission-limits.ts`,
  - non-RSVPed attendee is blocked from event photo uploads.
- Test helpers and QA scripts derive the API prefix from `API_VERSION` and default to `v1`; prefer `PLAYWRIGHT_API_BASE` only when you need a full explicit API base URL override.
- Workspace scripts now resolve Node 20 through `scripts/run-node20.sh` / `scripts/pnpm-node20.sh`; do not reintroduce transient `npx node@20` wrappers in package scripts because they have caused local path-resolution failures during typecheck and Playwright runs.
- Multi-step destructive test/setup scripts must use `scripts/with-target-env-shell.sh` so target-environment variables stay active across chained `&&` commands. Do not prepend only the first command in a chain with `with-target-env.sh`.

For local debugging, append Playwright args directly when needed, for example:

```bash
npx -y node@20 node_modules/@playwright/test/cli.js test --config=playwright.config.ts tests/e2e/domains/auth --project=chromium
```

## 10. Maintenance Rules

- Add new tests to the correct test-type bucket first, then domain bucket.
- Keep each spec focused on a user behavior or rule set.
- Reuse shared helpers only when it improves readability and consistency.
- Remove dead helpers/config/scripts when replacing a path; avoid layered legacy.
- Update this README whenever folder structure or command conventions change.
- Ensure local commands and CI command paths remain aligned.

## 11. Legacy Notes

The previous fragmented layout has been replaced:

- Removed legacy phase-based E2E structure (`phase1`, `phase3`).
- Removed parallel Playwright phase configs in favor of the main config + i18n contract config.
- Consolidated duplicated helper/fixture/setup paths into `tests/e2e/shared`.
- Relocated scenario/module specs into domain/journey/smoke/visual buckets.
- Added dedicated `tests/e2e/a11y` bucket for keyboard/label/focus smoke checks.
- Preserved manual QA plans under `tests/manual` as non-automated reference material.

Migration expectation for contributors:

- Do not add new tests to removed phase folders.
- Treat `tests/e2e/domains`, `tests/e2e/journeys`, `tests/e2e/a11y`, `tests/e2e/smoke`, and `tests/e2e/visual` as the canonical extension points.

## 13. `qa:seed` + `qa:browser-flow` Argument Reference

`qa:seed` is a convenience wrapper over `qa:browser-flow` that defaults to seed-safe behavior:

- `pnpm qa:create-data:seed:safe`
- `pnpm qa:create-data:seed:baseline:quick`
- `pnpm qa:create-data:seed:moderation`
- `pnpm qa:create-data:seed:moderation:quick`
- `pnpm qa:create-data:seed:pending`
- `pnpm qa:create-data:seed:quick`
- `pnpm qa:create-data:seed:all:quick`
- `pnpm qa:create-data:seed:all:quick:pending`
- `pnpm qa:seed -- --config <path>`

### Core CLI flags

- `--config <path>` JSON config file.
- `--headed` launch visible browser (`headless=false`) so you can watch step-by-step creation.
- `--headless <true|false>` explicit headless override.
- `--flow-mode <full|seed>` (alias: `--mode`)
- `--creation-only <true|false>` (seed defaults to true)
- `--admin-approve-created <true|false>`
- `--base-url <url>`
- `--web-port <port>`
- `--password <defaultPassword>`
- `--use-seeded-users <true|false>`
- `--reset-before-seed <true|false>` is ignored by the current seed runners.
- `--prompt-reset` is ignored by the current seed runners.
- `--uploads-dir <path>`
- `--runtime-dir <path>`
- `--screenshot-dir <path>`
- `--result-output-path <path>`

### User/media flags

- `--users <json>` config-preferred for multi-user payloads.
- Per-user config fields support:
  - `profileImagePath`
  - `portfolioImagePaths`
  - `allowProfileModerationBlock`
  - `allowPortfolioModerationBlock`
- `--img1 <path>` ... `--img5 <path>`

- `--portfolio-images <path1,path2,...>`
- `--opportunity-images <path1,path2,...>`
- `--event-cover-image <path>`

### Example: create one contest with arguments

```bash
pnpm qa:browser-flow -- \
  --flow-mode seed \
  --creation-only true \
  --contest-title "abc" \
  --contest-description "def" \
  --contest-deadline "2026-12-31" \
  --contest-banner-image "/absolute/path/to/banner.jpg" \
  --contest-resource-files "/absolute/path/to/ref-1.jpg,/absolute/path/to/ref-2.jpg"
```

Notes:

- Contest is created through admin login in the browser flow.
- Contest has `deadline` input; there is no explicit user-entered contest start date field in this flow.
- Contest resource uploads now use `resourceFilePaths` for one or more local files; the legacy singular `resourceFilePath` and `--contest-resource-file` alias remain accepted.
- Any omitted optional field falls back to defaults or empty values.

## 14. `qa:moderation-e2e` Argument Reference

### Supported CLI flags

- `--config <path>` JSON config file.
- `--mode <smoke|full>`
- `--run-id <id>`
- `--output-path <path>`

### Contest date/file behavior in current scripts

- `qa:browser-flow` contest creation uses:
  - `title`: argument-supported
  - `description`: argument-supported
  - `banner image`: argument-supported
  - `deadline`: argument-supported (`--contest-deadline`) with fallback default
  - `start date`: backend default (not passed by script)
  - `resource files`: argument-supported (`--contest-resource-files`; legacy singular alias still accepted)
- `qa:moderation-e2e` contest creation currently follows the same pattern (title/description/banner configurable; no explicit start/deadline/resource args).

### Event gallery upload lock behavior in current scripts

- `qa:browser-flow` now distinguishes between:
  - expected pre-event or RSVP-locked gallery states (`event-photo-upload-locked`)
  - true missing upload UI regressions (`event-photo-upload-unavailable`)
- Use the locked result as a product-state confirmation, not as a defect, when the event page explicitly says uploads unlock after the event starts or after RSVP.
